# 图像采集与人物分析模块设计

> 状态：已确认并进入 MVP 开发  
> 日期：2026-06-26  
> 负责人范围：门户页、拍摄授权、用户基础信息输入、摄像头引导、自动抓拍、图像质量检测、人物与当前穿搭分析、结构化结果交付  
> 下游契约：`BODY_PROFILE_CONTRACT.md` + `OUTFIT_PROFILE_CONTRACT.md`；HTTP API 交付 `analysis-handoff`

## 1. 结论

该模块适合作为 Dynamic AI Dress Room 的入口，并且能够在浏览器和 Google Cloud 环境中实现。

推荐的 Hackathon MVP 不是“用一张照片精准量体”，而是：

1. 通过站位框和姿态检测稳定获得一张可分析的全身正面照。
2. 使用用户输入的身高、体重作为视觉估算先验。
3. 分析当前穿搭、粗粒度体型和身体比例。
4. 对不可靠字段返回 `null` 或低置信度，而不是伪造精确结果。
5. 让用户确认或修正关键结果后，再交给推荐 Agent。

该定位既能形成有吸引力的现场体验，也不会让整个项目被高精度人体测量拖住。

已确认的第一版范围：

- 演示设备：普通笔记本电脑。
- 拍摄方式：单张正面照，保留手动快门兜底。
- 围度字段：近似估算，不可靠时返回 `null`。
- 结果交付：通过 HTTP API 给下游模块。
- 当前穿搭：独立输出 `outfit_profile`。

## 2. 与比赛要求的关系

图像采集模块是 Agent 的“感知入口”，而不是独立的 Agent。

完整演示应体现：

```text
采集用户状态
  -> 视觉分析工具生成结构化观察结果
  -> 造型 Agent 判断还缺少哪些信息
  -> Agent 查询库存并规划候选穿搭
  -> Agent 根据反馈重新调用工具和调整方案
```

建议采用：

- 浏览器端：摄像头、站位引导、轻量姿态检测和自动抓拍。
- Cloud Run API：会话、上传、分析任务与结果校验。
- Gemini 多模态：当前穿搭和粗粒度外观分析。
- Cloud Logging：记录拍摄失败原因、分析延迟、置信度和用户修正情况。
- GitHub Actions / Cloud Build：测试与部署。

这样可以同时支撑比赛强调的 Build、Operate 和 Deliver，而不只是展示一次模型调用。

## 3. 三种设备场景的共同流程

三个落地场景应共享同一套 Web 应用和 API，只通过设备配置改变交互：

| 场景 | 主要特点 | 推荐交互 |
|---|---|---|
| 试衣间镜面屏 | 距离和背景相对稳定 | 自动拍摄优先，允许正面与侧面 |
| 店铺入口大屏 | 人流复杂、隐私风险最高 | 必须先明确授权，只保留当前会话 |
| 店员 iPad | 摄像头可能由店员控制 | 同时提供自动和手动快门 |

第一版建议优先按“笔记本或大屏浏览器 + 单人正面照”实现，同时保持响应式布局兼容 iPad。

## 4. 推荐用户流程

### 4.1 页面流程

1. **门户页**
   - 解释功能和预期耗时。
   - 提供“开始体验”按钮。

2. **授权与数据说明**
   - 单独请求摄像头权限。
   - 明确照片用途、是否上传、保存时间和删除方式。
   - 用户主动确认后才能进入拍摄。

3. **基础信息输入**
   - 身高 `height_cm`
   - 体重 `weight_kg`
   - 性别呈现 `gender_presentation`
   - 年龄段 `age_range`

4. **摄像头引导**
   - 显示全身虚线框。
   - 实时显示“请后退”“头部未入框”“请保持不动”等提示。
   - 允许用户切换摄像头和退出。

5. **自动拍摄**
   - 条件持续满足一段时间后显示倒计时。
   - 抓取单张高质量画面。
   - iPad 和异常场景保留手动快门。

6. **质量检查**
   - 不合格时说明具体原因并重新拍摄。
   - 合格后上传分析。

7. **分析与确认**
   - 展示当前穿搭标签和中性的身体特征描述。
   - 允许用户纠正关键字段。
   - 不直接向用户展示带评价色彩的 `body_size` 标签。

8. **交付下游**
   - 输出版本化 JSON。
   - 通知推荐 Agent 进入需求沟通阶段。

### 4.2 拍摄状态机

```text
idle
  -> requesting_permission
  -> collecting_manual_profile
  -> camera_ready
  -> aligning
  -> stable
  -> countdown
  -> captured
  -> quality_check
      -> aligning          # 质量不合格
      -> analyzing         # 质量合格
  -> review
  -> confirmed
  -> handed_off
```

任何阶段都应支持 `cancelled` 和可恢复的 `error` 状态。

## 5. 自动抓拍判定

浏览器端可以使用 MediaPipe Pose 或同类轻量姿态模型。首版不需要把实时视频发到云端。

建议同时满足以下条件才开始倒计时：

- 画面中只检测到一个主要人物。
- 头部、双肩、髋部、膝盖和脚踝都在安全区域内。
- 人体中心位于虚线框中心附近。
- 人体占画面的比例处于合理范围。
- 肩部倾斜和身体转向在正面拍摄阈值内。
- 关键点平均置信度达到阈值。
- 最近若干帧的位置变化较小。
- 条件持续满足约 0.8 到 1.5 秒。

拍摄后还应检查：

- 图像分辨率。
- 清晰度或运动模糊。
- 曝光是否过暗或过亮。
- 人物是否被裁切。
- 是否出现多人。
- 是否存在严重遮挡。

所有阈值都应进入配置，而不是散落在 UI 代码中。

## 6. 分析职责拆分

### 6.1 浏览器端负责

- 摄像头权限与预览。
- 人体是否入框。
- 姿态稳定和自动倒计时。
- 基本图像质量检测。
- 必要时压缩图片。

### 6.2 后端负责

- 创建匿名会话和分析任务。
- 生成受控上传地址或接收上传。
- 调用 Gemini 多模态。
- 校验并规范化模型 JSON。
- 计算或补充置信度。
- 管理短期图片生命周期。
- 向下游返回统一结果。

### 6.3 Gemini 适合负责

- 当前服装单品类别。
- 主色、图案和材质观感。
- 当前穿搭风格。
- 粗粒度身体比例和体型分类。
- 图像是否足以支持某项判断。

### 6.4 Gemini 不应单独负责

- 从单张穿衣照片给出精确胸围、腰围或臀围。
- 精确判断服装尺码和真实合身程度。
- 未经用户输入推断年龄或性别。
- 输出缺乏依据却看似精确的小数测量值。

## 7. `BODY_PROFILE_CONTRACT.md` 字段能力评估

| 字段 | MVP 输出策略 | 稳定性 |
|---|---|---|
| `height_cm` | 用户输入 | 高 |
| `weight_kg` | 用户输入 | 高 |
| `gender_presentation` | 用户输入 | 高 |
| `age_range` | 用户输入 | 高 |
| `body_shape` | 照片粗分类；不足时 `null` | 中 |
| `body_size` | 根据输入与视觉综合分类；内部使用 | 中 |
| `proportions.shoulder_width` | 关键点与轮廓相对比例 | 中 |
| `proportions.waist_definition` | 受服装影响明显 | 中低 |
| `proportions.hip_width` | 关键点与轮廓相对比例 | 中 |
| `proportions.leg_to_torso` | 姿态关键点相对比例 | 中高 |
| `measurements.bust_cm` | 单张正面照不保证；允许 `null` | 低 |
| `measurements.waist_cm` | 单张正面照不保证；允许 `null` | 低 |
| `measurements.hip_cm` | 单张正面照不保证；允许 `null` | 低 |
| `measurements.shoulder_cm` | 可近似估算，必须带置信度 | 中低 |
| `measurements.inseam_cm` | 可近似估算，必须带置信度 | 中 |
| `measurements.foot_length_cm` | 默认 `null`，改由用户输入鞋码更合理 | 很低 |
| `skin_tone` | 可选；注意光照偏差 | 中低 |

建议修改现有契约：

- 允许所有照片提取字段为 `null`。
- 将测量值定义为 estimate，而不是 measurement truth。
- 为每一个测量字段提供独立置信度，不只提供聚合的 `measurements` 置信度。
- 增加 `analysis_warnings`，说明宽松服装、遮挡、光线或姿态造成的不确定性。
- 明确 `neutral` 对应的 `body_shape` 规则。
- 图片引用使用短期签名 URL 或内部对象 ID，不使用永久公开 URL。

## 8. 当前穿搭需要独立契约

`BODY_PROFILE_CONTRACT.md` 不包含当前穿搭，但后续“相似匹配”和风格推荐都依赖这部分数据。

建议独立输出：

```jsonc
{
  "outfit_profile": {
    "schema_version": "1.0",
    "overall_style": ["smart_casual"],
    "dominant_colors": ["navy", "white"],
    "items": [
      {
        "category": "top",
        "subcategory": "shirt",
        "color": "white",
        "pattern": "solid",
        "fit": "regular",
        "confidence": 0.91
      }
    ],
    "overall_confidence": 0.86,
    "analysis_warnings": []
  }
}
```

建议由团队另建 `OUTFIT_PROFILE_CONTRACT.md`，避免身体数据与服装识别相互污染。

## 9. 推荐接口边界

### 9.1 创建会话

```http
POST /api/sessions
```

返回匿名 `session_id` 和图片保存策略。

### 9.2 提交基础信息

```http
PUT /api/sessions/{session_id}/manual-profile
```

### 9.3 上传并分析

```http
POST /api/sessions/{session_id}/captures
Content-Type: multipart/form-data
```

返回：

```jsonc
{
  "analysis_id": "ana_123",
  "status": "processing"
}
```

### 9.4 查询分析状态

```http
GET /api/analyses/{analysis_id}
```

状态：

```text
queued | processing | needs_recapture | ready | failed
```

### 9.5 用户确认结果

```http
POST /api/analyses/{analysis_id}/confirm
```

请求包含用户修正字段。响应返回最终的 `body_profile` 和 `outfit_profile`，供下一个模块消费。

Hackathon MVP 若分析延迟较短，也可以先采用同步接口，但数据结构应保留 `analysis_id` 和状态，以便之后异步化。

## 10. 隐私与体验底线

- 在开启或上传摄像头画面前获得明确同意。
- 门口大屏不得因为检测到路人就自动拍摄。
- 默认不做人脸身份识别。
- 不把年龄和性别交给视觉模型猜测。
- 原始图片只在当前会话中使用，并设置自动删除时间。
- 日志中不得写入图片二进制、公开图片地址或可识别个人的数据。
- 给用户提供取消、重拍和删除当前会话数据的入口。
- 身体特征使用中性措辞，不在公开大屏展示敏感标签。

## 11. MVP 与后续版本

### MVP 必须完成

- 门户与授权页面。
- 四项用户输入。
- 摄像头预览和虚线站位框。
- 单人、全身入框、稳定姿态检测。
- 自动倒计时和手动兜底。
- 质量不合格时重拍。
- Gemini 结构化分析。
- JSON Schema 或等价的服务端校验。
- 结果确认与下游交付。
- Cloud Run 部署和基础日志。

### 有余力再做

- 正面与侧面双视角。
- 多设备配置。
- 用户修正结果反馈到评估数据集。
- 测量模型与 Gemini 结果交叉检查。
- 实时视频动作引导。

### 不进入本次 MVP

- 3D 人体重建。
- 医疗或裁缝级精确量体。
- 实时高精度视频换装。
- 人脸身份库。
- 对真实合身程度作保证。

## 12. 建议验收标准

| 类别 | 验收标准 |
|---|---|
| 授权 | 未同意时不启动采集和上传 |
| 拍摄 | 单人完整入框并稳定后可自动抓拍 |
| 失败恢复 | 多人、裁切、模糊等情况能提示并重拍 |
| 契约 | 输出通过 schema 校验，无法提取的字段明确为 `null` |
| 可解释性 | 包含置信度和警告信息 |
| 隐私 | 会话结束后原图按约定自动删除 |
| 下游对接 | 队友能使用固定接口取得最终 JSON |
| 部署 | 评委可通过公开 URL 完成一次流程 |
| 运维 | 能观察分析成功率、延迟和重拍原因 |

## 13. 需要团队立即确认的决策

这些问题会直接影响实现和接口，应在编码前确定：

1. MVP 是否接受所有围度为“近似估算”，并允许返回 `null`？
2. 是否同意修改 `BODY_PROFILE_CONTRACT.md` 的必填规则与置信度结构？
3. 是否新增独立的 `OUTFIT_PROFILE_CONTRACT.md`？
4. 第一优先设备是入口大屏、镜面屏，还是 iPad？
5. MVP 只拍正面，还是必须加入侧面照？
6. 下游通过 HTTP API、共享数据库还是同一前端状态接收结果？
7. 原始图片保存多久：分析后立即删除、会话结束删除，还是保留短期调试副本？
8. 用户是否需要在进入推荐阶段前确认和修正分析结果？

## 14. 推荐默认答案

如果团队希望尽快开工，建议采用以下默认方案：

- 单张正面照。
- 大屏浏览器为主、响应式兼容 iPad。
- 围度允许 `null`，不承诺精确尺码。
- 新增 `outfit_profile`。
- 用户确认后交付下游。
- HTTP JSON API 对接。
- 原始图片在会话结束后删除。
- MediaPipe 在浏览器端完成站位检测，Gemini 在后端完成语义分析。

这套范围能够保留项目最有吸引力的体验，同时将风险集中在可替换、可降级的分析环节。
